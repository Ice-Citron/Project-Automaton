# Help Request: Component Placement in MuJoCo + Isaac Lab Task Board

## Who This Is For
Gemini Deep Think / GPT-5.4 Pro — external help for the AIC (AI for Industry Challenge) project. Two Claude Code instances are working on this in parallel (one on MuJoCo, one on Isaac Lab) and both are stuck on the same component placement problem.

## Project Background
We're competing in the Intrinsic AI Challenge (AIC), where a UR5e robot arm inserts fiber optic cables (SFP, SC, LC connectors) into a task board. The competition evaluates in Gazebo, but we're building training environments in MuJoCo and Isaac Lab. The task board has been converted from Gazebo SDF → MuJoCo MJCF and from URDF → Isaac Lab USD.

The task board has 4 zones:
- **Zone 1**: Up to 5 NIC (Network Interface Card) cards on rails — WORKING PERFECTLY in MuJoCo
- **Zone 2**: Up to 2 SC (Subscriber Connector) optical ports on rails — WORKING PERFECTLY in MuJoCo
- **Zone 3-4**: Pick locations with mount fixtures (SFP mounts, SC mounts, LC mounts) holding plugs/modules — BROKEN, this is the problem

## The Exact Problem

### What's Working
The mount FIXTURES (the V-shaped cradle that screws to the rail) render correctly on the task board. We can see the SFP mount, SC mount, and LC mount in the right positions on the correct rails.

### What's Broken
The COMPONENTS that sit INSIDE the mounts (SFP transceiver modules, SC fiber plugs, LC fiber plugs) are misaligned. When we place a plug/module mesh at the mount body origin `pos="0 0 0"`, the plug appears at the wrong position and wrong orientation relative to the mount cradle. The plug should be nestled inside the V-cradle, but instead it's floating nearby or clipping through at the wrong angle.

### Why It's Broken
In the original Gazebo simulation, these plugs are NOT statically defined in the URDF/SDF scene files. Instead, the `aic_engine` (the competition's trial orchestrator) **dynamically spawns** the plugs at runtime and attaches them to the mounts via Gazebo-specific services. The MuJoCo and Isaac Lab environments don't have `aic_engine`, so we need to manually place the plugs.

The problem is: **we don't know the exact relative transform (position + orientation) between each mount's coordinate frame and the plug/module that sits inside it.** This transform is buried in the `aic_engine` runtime code, not in any XACRO/SDF/URDF file we can find.

**IMPORTANT NOTE:** These are plug-to-CABLE transforms, NOT plug-to-MOUNT transforms. They tell us how the plugs connect to the cable endpoints, not how they sit in the mount cradles. The mount-to-plug transform is what `aic_engine` computes at runtime when spawning components.

### MuJoCo OBJ Import Axis Issue (Detailed)
When we convert GLB → OBJ using trimesh and load in MuJoCo, the coordinate axes get permuted. We confirmed this with a controlled experiment:

**Test:** Created a box with known dimensions: X=0.10m (long), Y=0.02m, Z=0.05m. Wrote it as raw OBJ with vertex `(0.1, 0, 0)` explicitly in the file.

**Result:** MuJoCo loaded it as: X=0.02m, Y=0.05m, Z=0.10m. The axes were cyclically permuted: `MuJoCo(X,Y,Z) = file(Y,Z,X)`.

**Verification:** Checked actual vertex data:
- File vertex `(0.1, 0, 0)` centered to `(0.05, -0.01, -0.025)` appeared in MuJoCo as `(-0.01, -0.025, 0.05)` — confirming `(Y,Z,X)` permutation.

**Attempted fixes:**
1. Pre-rotating OBJ vertices before export — FAILED. Applied `(x,y,z)→(z,x,y)` transform to vertices, but MuJoCo showed the SAME result as without pre-rotation. The rotation appears to be applied AFTER MuJoCo's internal processing, not on the raw file data.
2. Applying `quat="0.5 0.5 0.5 0.5"` (120° around (1,1,1)/√3) on geom — FAILED. Made mounts lie flat and plugs pitch 90°. The actual axis mapping is more complex than a simple cyclic permutation for non-trivial meshes.

**Current theory:** MuJoCo may perform a data-dependent mesh reorientation during import (possibly related to inertia computation or convex hull analysis). This means the axis permutation may vary per mesh, making a universal correction impossible.

**For Isaac Lab:** This issue is specific to MuJoCo's OBJ loader. Isaac Lab uses USD format with Isaac Sim's URDF importer, which handles axis conventions properly. You should NOT have this problem.

## What We've Tried (Complete List with Results)

### Attempt 1: Place plug at mount origin (0,0,0)
**Action:** Added plug mesh geom inside mount body at `pos="0 0 0"` with no rotation.
**Result:** Plug appeared but completely misaligned — offset from cradle, wrong orientation. The plug and mount mesh origins are at different points in their respective geometries.

### Attempt 2: Adjust Z height
**Action:** Tried Z offsets of 0.005, 0.008, 0.01, 0.012, 0.015, 0.02, 0.025 for the mount body relative to the task board.
**Result:** Could get mounts at the right height on the board, but plugs inside mounts were still misaligned. The Z adjustment only affects the mount-to-board relationship, not the plug-to-mount relationship.

### Attempt 3: Use cable-end plug meshes directly on rails
**Action:** Instead of mount+plug combos, placed just the plug meshes (sc_plug_visual, lc_plug_visual, sfp_module_visual) directly on mount rails.
**Result:** Plugs appeared but showed their cable ferrules/strain-relief sticking out, which looked wrong for a static board. Also they were still at wrong orientation relative to the rail.

### Attempt 4: Convert and use actual mount OBJ meshes
**Action:** Converted LC mount and SC mount GLBs to OBJ using trimesh. Placed them on rails.
**Result:** Mount fixtures looked correct on the board! But when we nested plug meshes inside, the plugs were at wrong position/orientation relative to the mount.

### Attempt 5: Pre-rotate OBJ vertices to compensate for MuJoCo axis swap
**Action:** Identified MuJoCo's (Y,Z,X) permutation. Applied inverse rotation `(x,y,z)→(z,x,y)` to trimesh vertices before OBJ export using matrix `M=[[0,1,0],[0,0,1],[1,0,0]]`.
**Result:** FAILED. MuJoCo showed the same axis ordering regardless of pre-rotation. Confirmed with raw OBJ (bypassing trimesh) — the pre-rotation doesn't survive MuJoCo's import.

### Attempt 6: Apply correcting quat on mesh geoms
**Action:** Added `quat="0.5 0.5 0.5 0.5"` (120° around (1,1,1)/√3) to every mesh geom to undo the (Y,Z,X) permutation.
**Result:** FAILED. Mounts lay flat on the board instead of standing upright. Plugs pitched 90° forward. The quat overcorrected some axes and undercorrected others, suggesting the actual import rotation isn't a clean cyclic permutation for complex meshes.

### Attempt 7: Read XACRO/SDF for mount→plug transforms
**Action:** Read all mount XACROs, plug model SDFs, cable model SDF. Extracted every joint, link, and pose definition.
**Result:** Mount XACROs only define the fixture geometry (collision boxes, visual mesh). They do NOT define child links or joints for plugs. The cable SDF gives plug-to-cable transforms, not plug-to-mount transforms. The `aic_engine` handles the mount→plug attachment at runtime.

### Attempt 8: Compute mesh bounding boxes for offset estimation
**Action:** Loaded each GLB and OBJ in both trimesh and MuJoCo, compared bounding boxes to determine the axis mapping.
**Result:** Successfully identified that GLB and OBJ have consistent coordinates, but MuJoCo applies a (Y,Z,X) rotation on import. However, this knowledge alone wasn't enough to compute correct plug-in-mount positions because the plug orientation (which way is "up" and "forward") also gets scrambled.
